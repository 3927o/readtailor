# 阅读统计架构方案

## 背景

当前实现使用客户端累计 heartbeat：前端在正式阅读器内累加有效秒数、向前阅读秒数和向前字符数，后端按
`clientIntervalId` 幂等写入 `reading_sessions`，再聚合出全局和按书统计。

这个方案实现成本低，适合早期闭环，但长期会遇到几个结构性问题：

- 服务端难以独立校验阅读分类和字符推进量。
- 跨本地午夜、并发 heartbeat、永久删除书籍后的全局统计保留都容易产生错账。
- 客户端区域判断不够精确时，辅助内容可能污染个人阅读速度样本。
- 统计原始数据不够细，后续难以重建、审计或调整口径。

目标不是重写阅读器，而是把“阅读统计采集与聚合”升级成服务端可信、可重建、可演进的一层。

## 设计目标

- 只统计正式阅读器内的有效活动。
- 全局累计和连续天数在永久删除书籍后不倒退，也不暴露已删除书籍身份。
- 按书统计可以从原始活动重建。
- 个人阅读速度只来自正常连续向前阅读原文，不受跳转、回读、辅助内容和停留污染。
- 跨本地自然日的阅读能正确分桶。
- 网络重试、乱序到达和多标签页并发不会重复计数。

## 总体模型

阅读统计分三层：

```text
客户端观察层
  -> activity slice API
  -> 服务端校验与分类层
  -> 聚合查询层
```

客户端只提交它观察到的事实：活动时间段、起止位置、活动区域、会话 id、sequence 和本地时区。客户端不再作为阅读速度和字符推进的最终裁判。

服务端负责把观察值转成可信统计事实：校验时间跨度、去重、识别跳转/回读/向前读/辅助内容、计算原文字符推进量、跨午夜拆分自然日。

查询接口只读聚合表，不在请求时重新扫描大量原始事件。

## 核心数据

建议保留或新增这些概念：

- `reading_sessions`：一次进入正式阅读器的会话。记录用户、书籍、客户端 session id、时区、开始/结束时间和状态。
- `reading_activity_slices`：统计原始事实。以 `(user_id, client_session_id, sequence)` 幂等，记录起止时间、起止位置、活动区域、服务端分类、有效秒数、向前字符数。
- `reading_daily_book_stats`：按用户书籍和本地自然日聚合，记录有效秒数、速度样本秒数、向前字符数、最近阅读时间。
- `book_reading_stats`：按书累计缓存，记录累计有效秒数、速度样本、当前个人速度、最近阅读时间。
- `reading_daily_totals`：只按用户和本地自然日聚合，不关联书籍。用于今日、本周、累计和连续阅读天数，并在永久删除书籍后保留。

`reading_activity_slices` 和按书聚合可以随永久删除清理；`reading_daily_totals` 作为全局统计骨架保留。

## 客户端职责

前端 tracker 负责切分活动片段，而不是计算最终统计结果。

它应该提交：

- `client_session_id`
- `sequence`
- `slice_started_at`
- `slice_ended_at`
- `timezone`
- `start_position`
- `end_position`
- `activity_area`

`activity_area` 可以是 `original`、`assistance`、`reader_chrome` 等粗分类。它只是服务端分类的输入之一，不是最终统计结论。

前端仍然负责这些行为：

- 页面隐藏、离开、空闲、节点切换时结束当前 slice。
- 目录跳转和恢复滚动标记为非连续阅读。
- 不在试读页、书架、独立问 AI、统计页创建 reading activity。

## 服务端职责

服务端接收 slice 后，在同一事务内完成：

1. 校验 session 属于当前用户和当前 user-book。
2. 使用 `(user_id, client_session_id, sequence)` 幂等写入。
3. 拒绝负时间、未来时间、异常跨度和明显不合理的客户端输入。
4. 根据时区把 slice 拆到一个或多个本地自然日。
5. 根据 `reading_position_index` 计算位置差和字符推进量。
6. 分类为 `original_forward`、`original_reread`、`original_jump`、`assistance` 或 `stationary`。
7. 更新按书每日聚合、按书累计缓存和全局每日汇总。

只有 `original_forward` 的有效秒数和向前字符数进入个人速度样本。所有有效阅读活动都可以进入阅读时长。

## 删除与保留

软删除阶段保留所有阅读数据，便于恢复。

永久删除时：

- 删除 `reading_sessions`、`reading_activity_slices`、`reading_daily_book_stats`、`book_reading_stats`、阅读位置、已读节点、划线笔记等按书数据。
- 保留 `reading_daily_totals`。

这样全局累计时长和连续阅读天数不会倒退，同时无法从保留表判断用户读过哪本已删除书籍。

如果产品明确接受永久删除后仍保留按书明细，可以不删除 session。但这会改变隐私和数据保留语义，需要在 PRD 中明确。

## 迁移路线

建议分三步落地，避免一次性重写：

1. **加固当前 heartbeat**
   - 已完成部分输入校验、幂等键作用域、最近阅读时间口径和前端 flush 时机。
   - 继续补跨午夜切段和辅助内容不进入 `forwardSeconds` 的临时防线。

2. **引入 slice 表与双写**
   - 新增 `reading_activity_slices` 和按书聚合表。
   - 前端开始提交 slice；后端同时维护新聚合和旧响应字段。
   - 查询接口仍保持现有响应形状，降低前端改动面。

3. **切换查询来源并清理旧模型**
   - 统计查询改为读取新聚合表。
   - 删除或降级旧 cumulative heartbeat 逻辑。
   - 增加重建任务：可以从 `reading_activity_slices` 重建每日/按书聚合。

## 不建议的路线

不建议继续把所有口径都补在 cumulative heartbeat 上。这样短期能缓解问题，但服务端仍然无法可靠判断活动类型、跨天分桶和字符推进，后续每新增一个统计口径都需要继续绕当前模型打补丁。

也不建议把所有统计计算都放在前端。阅读统计属于长期数据资产，应该由服务端保存可验证的原始事实和可重建聚合。

## 开放问题

- 时区来源是否完全使用浏览器 IANA timezone，还是只接受客户端传来的 local date。
- `reader_chrome` 内的目录、设置、统计面板活动是否计入有效时长。
- 最大 slice 时长、空闲阈值、异常速度过滤阈值应配置化还是固定实现参数。
- 多设备同时阅读同一本书时，是否允许并行有效活动，还是需要服务端做重叠裁剪。
