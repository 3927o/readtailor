## 1. 产品定义

裁读是一个中文网页陪读产品。用户上传或从书架打开一本 EPUB 后，系统通过访谈了解用户
阅读这本书的目的、基础和阻碍，为用户生成一套本书处理方式，并通过试读确认后，在不修改
原文的前提下提供个性化导读、裁读注、节后助读和问 AI。

裁读不是摘要工具，也不替用户阅读。产品的核心价值是降低用户阅读一本具体书籍时的理解
阻力，使用户愿意开始并持续阅读原书。

### 1.1 首发产品目标

首发版本必须完成以下闭环：

```text
注册登录
  -> 获得预置书籍或上传 EPUB
  -> 完成每本书访谈
  -> 查看读前简报和处理方式
  -> 确认处理方式并生成三个试读片段
  -> 查看试读并最终采用
  -> 在阅读器中阅读原文和个性化辅助内容
  -> 通过划线或当前屏幕向 AI 提问
```

### 1.2 核心验证目标

MVP 主要验证：用户经过个性化配置后，是否愿意开始并持续阅读一本原本难以读下去的书。

产品必须记录足够的业务事件，以便后续计算以下行为：

- 注册后是否打开书架中的书
- 是否开始并完成本书访谈
- 是否确认处理方式并生成试读
- 是否查看完三个试读片段并最终采用
- 是否进入正式阅读
- 正式阅读期间的阅读节点、停留和问 AI 行为

首发不要求在产品内建设数据分析后台。

### 1.3 首发范围

- 公开互联网网页产品
- 中文产品界面
- 只处理中文 EPUB 书籍
- Google OAuth 登录
- 预置书籍自动进入新用户书架
- 用户上传 EPUB
- 每本书独立访谈和处理方式确认
- 三个代表性原文片段试读
- 在线阅读器
- 划线问 AI 和当前屏幕问 AI
- 跨设备同步书架、阅读进度、划线笔记、阅读统计和问答
- 书籍软删除与 30 天恢复

### 1.4 首发不做

- 支付、订阅、套餐和额度展示
- 公共书库、发现页、排行榜和书籍市场
- PDF、Word、扫描件、MOBI 等非 EPUB 输入
- 非中文书籍的专项处理
- EPUB 导出或重新打包
- 访谈跳过或推荐设置快速开始
- 无试读直接进入正式阅读
- 独立的处理参数面板、注释密度滑杆或逐节点强度设置
- 独立的“调整本书处理方式”入口
- 对单条 AI 内容点赞、点踩或举报
- AI 问答次数和使用额度限制
- 账户注销、数据导出和全账户删除流程
- 公共运营后台
- 邮件处理完成通知
- 默认防剧透功能
- 书签
- 不关联原文划线的独立笔记

## 2. 产品原则

### 2.1 原文不可变

规范化 HTML 是原书事实来源。导读、裁读注、节后助读、用户划线和问答均为独立附加数据，
不得写回、替换、删减或改写原文。

任何 AI 功能失败时，原文仍必须可以阅读。

### 2.2 先理解用户，再生成内容

每本书都必须完成独立访谈。长期画像不能代替本书访谈，用户也不能跳过访谈直接使用默认
策略。

### 2.3 初始处理方式的两次明确确认

用户必须完成两个不同的确认动作：

1. 确认文字处理方式，仅允许系统生成试读。
2. 查看试读后最终采用，才允许创建正式策略并进入阅读。

任何 Agent、后台任务或默认操作都不能代替用户确认。

正式阅读期间再次调整处理方式时，问 AI Agent 在判断需要调整后调用处理方式调整工具。工具
调用只创建待确认的调整建议，不能直接修改正式策略。前端在对应 AI 回复后展示确认卡；用户
可以直接确认，或在原问 AI 会话中反馈并让 Agent 修订后重新提交。不再要求查看节点效果预览。

### 2.4 辅助内容按实际价值生成

导读、裁读注和节后助读是系统支持的三种能力，不代表每本书、每个阅读节点或每个试读
片段都必须同时出现三种内容。

是否生成某一种内容由已确认的本书处理方式和当前原文决定：

- 处理方式可以规定全书不需要某一种辅助内容。
- 即使全书支持某一种内容，生成器也可以判断当前节点没有价值并返回空。
- 页面不得为缺失内容保留空卡片、空标题或“暂无内容”占位。
- 原文始终是阅读页面的主体。

### 2.5 确定性程序负责数据正确性

书籍哈希、状态迁移、阅读节点生成、block 枚举、精确锚点、权限和缓存必须由程序处理，不能
依赖模型自行保证。首发只做当前产品流程需要的状态校验，不建设通用版本门禁系统。

## 3. 核心对象

### 3.1 用户

通过 Google OAuth 创建。最小账户信息包括：

- 用户 id
- Google 身份标识
- 显示名称和头像
- 是否已完成初始长期画像
- 创建和最后登录时间

### 3.2 长期用户画像

跨书复用，只保存：

- `summary`：简短自然语言总结
- `knowledge[]`：用户可查看和纠正的知识背景条目
- `explanation_preferences[]`：用户可查看和纠正的解释偏好条目

注册后通过固定三题问卷生成初始版本。每本书访谈和问 AI 可以基于明确证据自动增加、修改
或删除画像条目。

问 AI 导致画像发生变化时，界面显示一次非阻塞提示，例如“已更新你的阅读画像”，不要求
用户确认，不打断当前对话。

首发不提供关闭自动更新、查看版本历史或复杂画像管理页面。产品保留当前画像和必要的变更来源，
用于解释结果和定位问题，不建设完整画像审计能力。

### 3.3 共享书籍包

共享书籍包按 EPUB 文件 SHA-256 精确复用，不包含用户数据。只有以下步骤全部完成后才能
进入 `ready`：

1. EPUB 解包成功。
2. Coding Agent 生成并执行 `normalize.py`。
3. `nb_check.py --baseline` 通过。
4. `reading_manifest.json` 确定性生成成功。
5. `assets/...` 路径及文件完整性校验通过。
6. 书籍分析 Agent 生成 `book_profile.json`。

书名、作者数组、语言、封面路径、标识符、出版社、出版日期、源文件名和 EPUB SHA-256 等
结构化元数据保存在共享书籍数据库记录中，供书架和业务查询使用。首发不创建独立的
`book_metadata.json`。规范化 HTML head 和 manifest document 中的元数据只是同一次提取结果
的只读快照，不能单独编辑或作为互相竞争的事实来源。

预置书籍应在加入用户书架前准备好共享书籍包。所谓“未经裁读处理”是指尚未为该用户进行
访谈、生成策略和个性化内容，而不是每个新用户都重新规范化同一个 EPUB。

### 3.4 用户书籍

表示某个用户书架中的一本共享书籍。它关联：

- 共享书籍包
- 用户书籍工作流状态
- 本书用户画像
- 读前简报
- 策略草稿和正式策略版本
- 试读版本
- 节点增强内容
- 阅读进度、划线笔记、有效阅读时长和按书阅读统计
- 问 AI 会话
- 软删除信息

### 3.5 本书用户画像

由本书访谈生成，至少描述：

- 用户为什么读这本书
- 用户已经掌握的相关知识
- 用户希望读到的深度或结果
- 用户最可能遇到的理解阻碍或放弃点
- 用户预计投入的方式或程度
- 其他直接影响本书处理方式的结论

### 3.6 本书处理方式

同时包含：

- 面向用户展示的自然语言说明
- 程序消费的结构化策略

结构化策略至少定义：

- 处理目标和表达原则
- 是否启用导读，以及导读要解决的问题
- 是否启用裁读注，以及注释重点和明确排除项
- 是否启用节后助读，以及助读目标
- 当前策略版本

策略是全书层面的原则，不保存逐节点强度。节点生成器仍可根据节点实际内容返回空结果。

### 3.7 阅读节点、片段和 block

正式阅读节点遵循 `reading_contract.md` 的 `section_id + segment` 模型。

`reading_manifest.json` 还保存从完整 `section[data-type]` 树确定性生成的 `outline`，供连续
滚动阅读器展示目录。outline 包含没有自身阅读节点的纯分组 section，并用 `first_node_order`
指向该 section 自身或首个后代的第一个阅读节点。它是产品导航索引，不取代规范化 HTML，
也不伪造原 EPUB 的 TOC。

试读片段是阅读节点内部一段连续的原文范围，使用以下位置定位：

```json
{
  "section_id": "ch-001",
  "segment": 1,
  "range": {
    "start": { "block_index": 3, "offset": 0 },
    "end": { "block_index": 7, "offset": 42 }
  }
}
```

试读范围可以跨同一节点内的多个 block，但不能跨阅读节点。范围必须在生成任务创建前由程序
校验，并可以从不可变规范化书籍包中确定性恢复原文。

### 3.8 裁读资格

不是所有阅读节点都允许生成裁读内容。阅读节点服务于原文加载、位置、进度、划线、笔记和
问 AI；裁读资格只决定系统是否可以为该节点生成导读、裁读注和节后助读。

首发只有同时满足以下条件的节点具有裁读资格：

- `region = bodymatter`
- 节点所有者的 `data_type` 为 `chapter`、`section` 或 `subsection`
- 节点包含可阅读的实质原文，而不是空节点、纯分隔符或纯媒体

以下内容无论文本多少，均不生成裁读内容：

- `frontmatter` 和 `backmatter` 中的全部节点
- `part` 自身直接拥有的正文 segment
- 前言、他序、导论、序章、后记、结语、附录
- 扉页、版权与出版信息、献辞和题记
- 摘要、致谢、书目、术语表和索引
- 原书 notes 区及其他功能型内容
- 顶层区域直接拥有的 segment

对应 `data_type` 包括但不限于 `preface`、`foreword`、`introduction`、`prologue`、
`epilogue`、`appendix`、`afterword`、`titlepage`、`colophon`、`dedication`、`epigraph`、
`abstract`、`acknowledgments`、`bibliography`、`glossary` 和 `index`。

reading manifest 必须为每个阅读节点保存确定性的资格结果：

```json
{
  "section_id": "sec-001",
  "segment": 1,
  "tailoring_eligible": true,
  "exclusion_reason": null
}
```

`tailoring_eligible` 由 manifest 生成程序按独立的版本化规则计算，书籍分析 Agent 和个性化
策略只能读取，不能把明确排除的节点改成可裁读节点。没有资格的节点不创建裁读生成任务，
也不调用模型返回空结果。

没有裁读资格不影响用户阅读原文、打开原书注、划线、写笔记或问 AI。

### 3.9 共享书籍分析

`book_profile.json` 是紧凑的全书级 AI 语义分析，不是原文结构或位置索引。它至少包括：

- 全书概要和结构特点
- 核心问题和主要主题
- 主要阅读门槛
- 供访谈、策略和内容生成使用的全书级阅读建议
- 约 9–15 个共享试读候选位置

试读候选只保存可解析且具有裁读资格的 `section_id + segment`、候选特征和简短理由。它不
直接决定某个用户最终看到的三个片段；访谈与处理方式 Agent 必须结合本书用户画像读取候选
原文后再决定。

首发不在 `book_profile` 中保存覆盖全部 reading manifest 节点的摘要、内容类型或难度映射，
也不建设完整节点级 AI 索引。未来如确有需要，可以增加独立、可懒生成和缓存的
`node_analysis`，但它不能成为共享书籍包进入 ready 的前置条件。

### 3.10 阅读记录与统计

阅读记录属于用户书籍数据，不进入共享书籍包。系统至少保存：

- 当前稳定阅读位置和全书进度
- 已读节点
- 每次有效阅读 session 的起止时间和有效阅读秒数
- 按自然日聚合的阅读时长
- 每本书累计阅读时长和近期阅读速度
- 最近阅读时间

首发阅读统计至少展示：

- 今日阅读时长
- 本周阅读时长
- 累计阅读时长
- 连续阅读天数
- 每本书的累计阅读时长、全书进度和预计剩余阅读时间

连续阅读天数按用户本地自然日计算。某一天存在有效阅读记录即计为一个阅读日；只有打开阅读器、
但没有形成有效阅读时间的日期不计入连续阅读。

阅读统计不改变书籍主工作流状态，也不影响原文、策略和节点增强内容。

## 4. 信息架构

首发至少包含以下页面或视图：

1. Google 登录页
2. 初始长期画像页
3. 书架页
4. EPUB 上传与处理进度页
5. 单本书访谈页
6. 读前简报与处理方式确认页
7. 试读片段页
8. 正式阅读器
9. 问 AI 侧边面板或移动端独立视图
10. 最近删除页
11. 简版长期画像查看与纠正页
12. 阅读统计页或统计视图

可采用任意 Web 路由结构，但刷新页面和直接打开 URL 后必须能根据业务数据库恢复到正确
状态，不能依赖前端内存中的上一步数据。

### 4.1 全局导航

登录后的全局导航至少提供：

- 返回书架
- 查看“我的阅读画像”
- 查看阅读统计
- 当前账户菜单
- 最近删除入口
- 退出登录

阅读器内应弱化全局导航，优先保留阅读空间，但必须能返回书架。

### 4.2 全局状态要求

所有异步页面必须提供：

- 明确的加载状态
- 可理解的失败状态
- 在允许重试时提供重试按钮
- 防止重复提交的禁用状态
- 刷新页面后恢复任务状态的能力

产品界面不得展示模型思维过程、内部 prompt、对象存储路径、沙箱路径或原始脚本日志。

用户在访谈、处理方式确认、试读生成或试读查看期间返回书架、刷新页面或稍后重新打开书籍，
系统必须保留已提交答案、当前策略版本和当前流程状态，并恢复到原阶段。首发不提供“重新开始
整本书配置”按钮，也不能因为用户离开页面自动重置流程。

## 5. 注册与初始画像

### 5.1 登录页

页面只有一个主要动作：使用 Google 登录。

登录成功后：

- 新用户进入初始长期画像。
- 已完成初始画像的用户进入书架。
- 用户原本通过受保护链接进入时，登录后可恢复目标页面；若目标状态无权访问则回到书架。

### 5.2 初始长期画像

初始画像使用固定三题问卷，不使用 Agent 动态提问。每次展示一题，顶部显示 `1/3`、`2/3`、
`3/3`。用户可以返回修改已填写的问题，第三题提交后一次性生成画像。

#### 5.2.1 知识背景

问题：

> 下面哪些领域是你比较熟悉的？

允许多选，最多 3 项：

- 文学与艺术
- 历史、哲学与社会科学
- 商业、经济与管理
- 数学、自然科学与工程
- 计算机与互联网
- 没有特别熟悉的领域

附带可选文字输入：

> 也可以补充更具体的领域，例如“古希腊哲学”“机器学习”。

选择“没有特别熟悉的领域”时自动取消其他选项；选择其他选项时自动取消“没有特别熟悉的
领域”。

#### 5.2.2 解释方式

问题：

> 遇到陌生概念时，你更喜欢怎样的解释？

允许多选，最多 2 项：

- 先用通俗的话说明，再介绍准确概念
- 多用具体例子或类比
- 从定义和逻辑关系一步步推导
- 直接给简洁结论，需要时再展开
- 没有固定偏好，根据内容决定

附带可选文字输入：

> 还有其他解释偏好吗？

选择“没有固定偏好”时自动取消其他选项，规则与第一题相同。

#### 5.2.3 背景补充深度

问题：

> 阅读不熟悉领域的内容时，你希望默认补充多少背景？

单选：

- 只补充理解当前内容必需的信息
- 适当说明来龙去脉，但不要偏离原书太远
- 尽量系统地补齐相关基础
- 根据内容决定

界面可以预选“适当说明来龙去脉，但不要偏离原书太远”，但用户必须主动提交第三题才能完成
问卷。

#### 5.2.4 画像生成

固定选项通过程序内的版本化映射表直接转换为 `knowledge` 和
`explanation_preferences`，不调用模型。`summary` 使用固定模板生成，保证注册流程不依赖 AI
可用性。

示例：

```json
{
  "summary": "用户熟悉计算机与互联网。解释陌生概念时，适合先用通俗表述，再引入准确术语，并适当补充必要背景。",
  "knowledge": [
    "熟悉计算机与互联网"
  ],
  "explanation_preferences": [
    "解释陌生概念时，优先使用通俗表述，再引入准确术语",
    "阅读陌生领域时，适当补充必要背景，但避免过度偏离原书"
  ]
}
```

可选自由文字使用一次普通模型调用做受限提取，不使用 Agent。模型只允许输出知识背景和解释
偏好条目，不得生成兴趣、阅读习惯、性格或普遍阻碍。提取失败时忽略自由文字，仅使用固定
选项完成注册，不能阻塞用户进入书架。

系统同时保存：

- 问卷 schema 版本
- 用户原始选择和自由文字
- 选项映射版本
- 自由文字提取状态及模型配置版本
- 最终 `reader_profile` 版本

要求：

- 用户不能跳过整个初始画像。
- 前两题必须至少选择一项，第三题必须选择一项。
- 最后一个问题提交后生成并保存 `reader_profile`。
- 生成失败时保留用户答案并允许重试，不能要求重新回答。
- 完成后把所有预置书籍加入用户书架，再进入书架页。
- 重复执行画像生成和预置书籍初始化必须幂等，不能产生重复画像版本或重复书籍。

### 5.3 查看和纠正长期画像

账户菜单提供“我的阅读画像”。页面只展示当前 `summary`、知识背景条目和解释偏好条目。用户
可以逐条修改或删除，并可新增一条自由文字知识背景或解释偏好；不展示版本历史、证据详情或
复杂开关。保存后生成新的画像版本，后续访谈、试读和节点生成使用新版本，已经发布的历史
裁读内容不被静默改写。

## 6. 书架

### 6.1 书籍列表

书架展示当前用户所有未删除书籍。每本书至少展示：

- 封面；没有封面时使用统一占位
- 书名
- 作者；未知时不显示空字段
- 当前产品状态文案
- 正式阅读中的全书进度
- 正式阅读中的预计剩余阅读时间
- 最近阅读时间

默认按最近操作时间倒序排列。首发不要求搜索、筛选和自定义排序。

### 6.2 状态与点击行为

| 内部状态 | 书架文案示例 | 点击后进入 |
|---|---|---|
| `on_shelf` | 开始裁读 | 本书访谈 |
| `interviewing` | 继续访谈 | 本书访谈当前问题 |
| `strategy_review` | 确认处理方式 | 读前简报与处理方式页 |
| `trial_generating` | 正在生成试读 | 试读生成进度页 |
| `trial_generation_failed` | 试读生成失败 | 失败与重试页 |
| `trial_review` | 查看试读 | 当前试读片段页 |
| `active_reading` | 继续阅读 | 上次阅读位置 |
| 共享书籍处理中 | 正在处理书籍 | 上传与处理进度页 |
| 共享书籍失败 | 该版本暂时无法处理 | 失败详情页 |

### 6.3 添加书籍

书架提供明确的“上传 EPUB”动作。只接受 `.epub` 文件。

文件选择后必须先校验：

- 文件存在且非空
- 扩展名为 `.epub`
- 服务端能够识别为 ZIP/EPUB；前端扩展名校验不能代替服务端校验

首发不规定文件大小和书籍数量限制。系统仍需对异常大文件提供技术性保护，具体阈值配置化，
不在普通界面展示为套餐额度。

### 6.4 删除

每本书的菜单提供“从书架删除”。确认后：

- 设置 `deleted_at` 和 `purge_after`
- 保留删除前 `workflow_status`
- 从普通书架消失并进入最近删除
- 暂停或取消该用户未完成的访谈、试读和节点生成任务
- 不取消共享书籍规范化任务

## 7. EPUB 上传与规范化

### 7.1 上传流程

1. 客户端上传 EPUB。
2. 服务端计算 SHA-256。
3. 已存在 `ready` 共享书籍包时，直接创建用户书籍并进入 `on_shelf`。
4. 已存在进行中的同哈希任务时，复用该共享任务并订阅进度。
5. 未命中时创建规范化任务。
6. 共享书籍包进入 `ready` 后，用户可开始本书访谈。

在共享书籍包 `ready` 前，不允许打开原始 EPUB 阅读，也不允许开始访谈。

### 7.2 处理进度

页面展示经过产品化映射的阶段：

- 正在上传
- 正在检查书籍文件
- 正在整理书籍结构
- 正在验证正文、图片和注释
- 正在建立阅读索引
- 正在分析书籍
- 处理完成

进度可以是阶段式，不要求提供不准确的百分比。刷新或跨设备打开后必须恢复当前阶段。

### 7.3 失败

解包、规范化、验证、索引或分析失败统一显示：

> 该版本暂时无法处理。你可以重试，或上传另一个版本的 EPUB。

页面提供：

- 重试当前版本
- 返回书架
- 上传另一个版本

首发不设置产品层面的重试次数限制，不提供人工处理入口，也不向用户展示内部校验报告。
后台记录失败阶段、错误类型和必要的校验报告引用，足以定位并重试本次处理即可。

## 8. 每本书访谈

### 8.1 进入条件

共享书籍包必须为 `ready`，且用户书籍尚未进入正式阅读。首次打开 `on_shelf` 书籍时进入
访谈。

用户最终采用试读对应的处理方式前，不能进入完整阅读器，也不能绕过访谈、处理方式确认和
试读流程阅读整本纯原文。此阶段可见的原文仅限三个试读片段。

### 8.2 页面结构

页面至少包含：

- 书名和封面的小型上下文标识
- 当前问题
- 2–5 个选项
- 自由文字补充输入
- 提交并继续按钮
- 简洁的进度表示，例如“第 3 问，最多 7 问”
- 返回书架

每次只展示一个问题。已提交答案不可在当前流程中通过浏览器后退静默改写；如需允许修改，
必须作为明确的重新回答操作并重新计算后续状态。MVP 可以不提供修改历史答案的入口。

### 8.3 提问规则

- 总问题数不超过 7 个。
- 信息足够时可以提前结束。
- 不询问长期画像中已经足够明确的信息。
- 问题必须直接服务于这本书的处理方式。
- 不能跳过整个访谈，也不能选择默认处理方式直接进入下一阶段。

### 8.4 完成访谈

Agent 判断信息足够后，在一次可恢复的后台流程中生成：

- `book_reader_profile`
- 可选 `reader_profile_patch`
- 个性化读前简报
- 用户可读的处理方式
- 结构化策略草稿

全部保存成功后进入 `strategy_review`。中途失败时保留答案和已成功产物，重试必须幂等。

---

## 9. 读前简报与处理方式确认

### 9.1 页面内容

页面按以下顺序展示：

1. 书名和封面
2. 个性化读前简报
3. “我们会怎样陪你读这本书”的自然语言处理方式
4. 自由文字反馈输入
5. 主按钮“处理方式没问题，生成试读”

读前简报应帮助用户理解：

- 这本书整体在做什么
- 全书大致如何展开
- 阅读前需要知道或临时补齐什么
- 对该用户而言哪些内容值得关注
- 哪些困难是正常的，以及如何进入阅读

处理方式必须用用户能理解的语言说明实际行为，不能展示 JSON、模型参数、prompt 或复杂设置。

### 9.2 反馈调整

用户可以提交自由文字反馈。每次有效反馈：

- `adjustment_count + 1`
- Agent 根据反馈生成新的文字处理方式和结构化策略草稿
- 旧草稿保留但不再是当前版本
- 页面展示新版本，等待用户再次确认

除非实现方判断反馈改变了本书画像，否则首发不要求重新生成读前简报。

### 9.3 第一次确认

用户点击“处理方式没问题，生成试读”后：

- 当前 `strategy_draft_version` 标记为 `approved_for_trial`
- 禁止继续编辑该版本
- Agent 选择三个试读片段
- 创建新的 `trial_revision`
- 系统开始生成三个试读结果
- 用户进入试读生成进度页

第一次确认不创建正式 `strategy_version`，也不能开始正式阅读。

### 9.4 调整上限

处理方式反馈和试读反馈共用 `adjustment_count`，初始版本不计数，最多 5 次。

达到 5 次后：

- 隐藏反馈输入框、反馈按钮和其他调整入口
- 保留当前阶段的确认或退出操作
- 用户仍可确认最后策略、生成最后一轮试读并最终采用

## 10. 试读片段

### 10.1 试读定义

试读单位是阅读节点内的一段连续原文片段，不是完整阅读节点。三个片段分别保存精确的
`section_id + segment + block range`，并使用与正式阅读相同的内容生成和锚点校验链路。

### 10.2 片段选择

访谈与处理方式 Agent 必须选择恰好三个不同的试读片段，原则上分别代表：

- 进入本书时的理解门槛
- 全书典型内容和表达方式
- 较高难度或最能体现处理方式价值的内容

约束：

- 每个片段必须位于一个 `tailoring_eligible = true` 的阅读节点内。
- 每个片段是同一节点内连续的 block range。
- 三个片段不能重叠。
- 不选择目录、版权页、纯图片页或没有实质正文的区域。
- 片段必须包含足够上下文，可以单独试读。
- 首发不做防剧透限制，允许选择中后段片段。
- 选择理由写入内部数据和日志，不要求展示给用户。

片段长度应配置化。初始实现建议以能够在一个页面内完成试读、同时足以体现导读和注释效果
为准，不在 PRD 中固定字符数。

### 10.3 生成一致性

试读和正式阅读必须共用：

- 同一个生成脚本
- 同一套 prompt 模板
- 同一个模型调用适配层
- 同一个输出 schema
- 同一套 quote 到 UTF-16 range 的锚点解析
- 同一套结果校验、重试和缓存逻辑

生成器增加明确的输入范围：

- `generation_scope = trial`：输入试读片段 range，只允许使用已批准的策略草稿。
- `generation_scope = formal`：输入完整阅读节点，只允许使用正式策略版本。

`generation_scope` 只控制输入范围、允许引用的策略类型和缓存隔离，不能使用演示专用 prompt、美化逻辑
或降低质量标准。

生成器所需的相邻上下文使用上一个可裁读节点末尾和下一个可裁读节点开头的受限原文，不
依赖预先生成的节点摘要。截取 block 数或字符数由程序的版本化配置决定，模型不能自行扩大
相邻原文范围。

### 10.4 生成结果

每个片段的固定输出仍为：

```json
{
  "guide": null,
  "annotations": [],
  "after_reading": null
}
```

根据处理方式和片段实际内容，三类结果均允许为空。若三类结果全部为空，说明该片段无法展示
处理方式的价值，整个片段结果视为不合格，应重新选择片段或重新生成，不能向用户展示纯原文
作为试读结果。

### 10.5 生成状态

三个片段使用同一个 `trial_revision` 和 `strategy_draft_version`，可以并行生成。

- 全部成功后才进入 `trial_review`。
- 任一片段耗尽技术重试后，整轮进入 `trial_generation_failed`。
- 不允许展示部分成功结果。
- 用户重试同一轮不增加 `adjustment_count`。
- 重试可以复用片段选择，也可以在 Agent 判断原片段不适合时重新选择，但必须创建新的
  `trial_revision`。

### 10.6 试读页面

三个片段逐个查看，不在同一屏并排展示。

页面至少包含：

- “试读 1/3”进度
- 片段所在的章节路径，避免用户误以为三个片段连续
- 当前片段的导读；有内容时显示
- 当前片段原文
- 锚定在原文位置的裁读注；有内容时显示
- 当前片段的节后助读；有内容时显示
- 上一个和下一个片段导航
- 第三个片段后的最终采用操作
- 自由文字反馈入口；未达到调整上限时显示

用户可以返回前一个片段重复查看。最终采用按钮只在用户已经至少打开过三个片段后可用。
“打开过”通过服务端记录片段查看事件，不依赖单次前端会话。

### 10.7 试读反馈

用户在任意试读片段提交反馈后：

1. 当前整个 `trial_revision` 立即标记为 `superseded`。
2. 已完成和尚未完成的三个片段任务及结果全部失效。
3. `adjustment_count + 1`。
4. Agent 根据反馈生成新的文字处理方式和结构化策略草稿。
5. 用户返回处理方式确认页。
6. 用户必须重新确认处理方式，系统才生成新一轮三个片段。

旧试读结果不能继续显示、确认或恢复为当前版本。

### 10.8 最终确认

用户点击“采用这个处理方式并开始阅读”时，服务端必须原子校验：

- 用户书籍状态为 `trial_review`
- `trial_revision` 是当前版本且未失效
- 三个片段全部成功并已发布
- 三个片段都已被用户打开过
- `strategy_draft_version` 是当前已批准草稿
- 当前草稿与三个片段生成时使用的版本完全一致

校验通过后：

- 将草稿复制或提升为新的正式 `strategy_version`
- 用户书籍进入 `active_reading`
- 创建第一个正式阅读节点和后续三个节点的生成任务
- 进入正式阅读器

重复点击必须幂等，不能创建多个正式策略版本。

## 11. 正式阅读器

### 11.1 页面布局

正式阅读器采用连续滚动。阅读节点是加载、定位和生成单位，但用户看到的是按 manifest 顺序
连续衔接的原文，不使用“一次只显示一个节点”的翻页模式。

桌面端至少包含：

- 可展开或收起的目录
- 阅读正文区域
- 全书进度、当前目录位置和预计剩余阅读时间
- 书籍菜单
- 问 AI 入口

移动端可以使用抽屉、底部面板或独立视图承载目录和问 AI，但功能必须完整。

书籍菜单至少提供：

- 查看读前简报
- 查看当前处理方式
- 阅读设置
- 返回书架

### 11.2 内容顺序

每个正式阅读节点按以下顺序展示：

1. 节点导读；存在时显示，默认展开，可收起
2. 原文
3. 节后助读；存在时显示，默认展开，可收起

如果导读或节后助读为空，对应区域完全不渲染。

原书注和裁读注都在相应原文位置提供标记。点击后在靠近原文的位置展开，且必须清晰标注：

- 原书注
- 裁读注

两种注释不得使用完全相同且无来源提示的视觉表现。

### 11.3 节点加载和提前生成

用户最终采用策略后，不等待整本书生成完成：

- 第一个正式阅读节点可以显示原文后，立即允许进入阅读器。
- 后台始终使当前节点及后续三个节点处于 `ready`、`generating` 或已排队状态。
- 用户跳转到其他节点时，提高目标节点及其后三个节点的优先级。
- 当前节点增强内容未完成或失败时，先显示纯原文，并在内容就绪后更新附加层。
- 更新附加层不能改变用户当前滚动位置。

上述生成和排队只作用于 `tailoring_eligible = true` 的节点。计算“后续三个节点”时跳过没有
裁读资格的节点，但这些节点的原文仍按正常阅读顺序加载和展示。

正式阅读后，用户可以通过目录任意跳转到全书任何位置，包括尚未读到的后文，不设置顺序
解锁。目标位置的裁读内容尚未生成时立即显示纯原文，并在后台提高相应生成任务的优先级。

### 11.4 已读判定

只要用户在一个阅读节点发生有效停留，该节点就标记为已读。

“有效停留”的默认实现规则：

- 节点原文实际进入可视区域；并且
- 页面处于可见状态。

满足以上条件即标记已读，不设置最短停留秒数。单纯预加载节点或页面在后台不可见时不能
标记已读。

节点一旦标记已读，不因用户回退、清除前端缓存或策略更新而变回未读。

### 11.5 阅读位置和进度

最后阅读位置保存到：

```text
section_id + segment + block_index + UTF-16 offset
```

保存要求：

- 滚动阅读期间节流保存
- 切换节点、页面隐藏、返回书架时立即保存
- 跨设备打开后恢复到最近一次成功保存的位置
- 位置失效时回退到同一节点首个有效 block，再失败则回退到最近有效节点

全书进度只按 `reading_manifest` 中的原文位置计算，不计导读、裁读注和节后助读。界面同时
显示全书百分比和当前目录节点。

### 11.6 阅读设置

首发阅读器必须提供基础阅读设置：

- 字号
- 行距
- 正文内容宽度
- 浅色、深色和跟随系统主题

设置使用即时预览并跨设备同步。设置只影响表现层，不能改变 block 枚举、标准文本、offset
或阅读进度。

首发可以不提供字体选择、翻页模式、复杂主题和自定义 CSS。

### 11.7 划线和笔记

用户可以选择同一阅读节点内、可跨 block 的连续原文。选择后提供：

- 问 AI
- 添加或编辑笔记
- 保存划线

首发只支持基于划线的笔记：用户必须先选择一段原文，才能为该划线添加笔记。产品不提供书签，
也不提供不关联原文划线的独立笔记。

划线和笔记使用 `reading_contract.md` 定义的 range。一个划线可以没有笔记，也可以关联一条用户
笔记；用户可以编辑或删除笔记，也可以删除整个划线。删除笔记时保留划线；删除划线时同时删除其
当前笔记，但不能删除由该划线发起的历史问 AI 会话。

### 11.8 有效阅读时间

系统只统计用户实际阅读期间的有效时间。满足以下条件时才累计：

- 用户位于正式阅读器。
- 页面处于前台可见状态。
- 用户近期存在滚动、触摸、点击、键盘或阅读位置变化等活动。

页面进入后台、窗口失去活动状态或用户持续无操作达到空闲阈值后暂停计时。用户恢复活动时开始新的
有效时间区间，不能把整段空闲时间补记为阅读时间。

客户端在阅读期间周期性上报 heartbeat，并在节点切换、页面隐藏、返回书架和正常退出时立即提交。
网络重试不能重复累计同一时间区间。heartbeat 周期和空闲阈值由实现配置，不能影响上述产品口径。

有效阅读时间包括用户在正式阅读器内阅读原文、导读、原书注、裁读注和节后助读的时间。试读页、
问 AI 独立视图、书架、阅读统计页、其他全局页面和其他书籍的时间均不计入当前书籍。

### 11.9 阅读统计

阅读统计提供全局和按书两个层级。

全局统计至少包括：

- 今日有效阅读时长
- 本周有效阅读时长
- 所有书籍累计有效阅读时长
- 当前连续阅读天数

按书统计至少包括：

- 该书累计有效阅读时长
- 最近阅读时间
- 当前全书进度
- 预计剩余阅读时间

阅读时长使用适合阅读的分钟/小时格式展示。统计更新可以有短暂延迟，但用户结束一次有效阅读后，
再次进入统计视图必须看到已提交的数据。

### 11.10 预计剩余阅读时间

剩余阅读时间按书计算，因为不同书籍的语言、难度、排版和用户阅读方式不同。计算使用：

```text
该书剩余原文字符量 / 用户阅读该书的有效阅读速度
```

剩余原文字符量基于 `reading_manifest` 和当前稳定阅读位置，只计算原文，不把导读、注释和节后助读
的字符数加入剩余工作量。

该书有效阅读速度只使用用户正常向前阅读原文的时间和对应字符推进量估算。回读、目录跳转、停留
不动、阅读导读/注释/节后助读的时间仍可计入阅读时长统计，但不进入阅读速度的分子或分母；目录
大幅跳转不能直接当作用户读完了中间内容。

当该书积累的数据不足时，使用按语言配置的默认阅读速度，并统一展示为近似值，例如“预计还需约
2 小时”。积累足够的有效阅读样本后自动改用该书的个人阅读速度。样本阈值、异常速度过滤和默认
速度属于实现参数，不在普通界面展示。

## 12. 问 AI

### 12.1 入口

支持两种入口：

1. 划线问 AI：初始上下文为划线范围、所在段落和当前节点。
2. 直接问 AI：没有划线时，初始上下文为当前屏幕覆盖的 block 范围和当前节点。

每个新问题创建一个独立会话。同一问题中的追问继续使用该会话。一本书不提供单一无限长
聊天会话。

### 12.2 对话界面

界面至少包含：

- 发起问题时的原文上下文，可折叠
- 用户问题和 AI 回答
- 追问输入框
- 生成中、失败和重试状态
- 返回原文位置的动作

从划线发起的问题必须能定位回原始 range。从当前屏幕发起的问题至少能定位回原始节点和
屏幕 block 范围。

首发不强制展示可点击原文引用，但回答不得伪装成原书原文。

### 12.3 检索范围

问 AI Agent 可以按需：

- 读取当前和相邻节点
- 查看全书结构
- 搜索全书
- 读取指定节点
- 读取原书注释
- 读取长期画像、本书画像和当前正式策略

允许检索用户尚未读到的后续内容，不提供默认防剧透限制。

### 12.4 更新长期画像

当对话提供了明确、稳定的用户背景或解释偏好证据时，Agent 可以调用工具更新长期画像。

- 不需要用户确认。
- 更新后显示非阻塞提示。
- 保存变更前后的画像、证据会话 id 和变更原因，便于解释本次更新。
- 不能把一次临时困惑直接升级为长期偏好。

### 12.5 调整本书处理方式

问 AI Agent 不能直接修改正式策略。当 Agent 根据当前对话判断需要调整本书处理方式时，调用
处理方式调整工具。工具创建待确认的 `strategy_change_proposal` 和候选
`strategy_draft_version`，前端在对应 AI 回复之后展示确认卡。

正式阅读阶段不再生成或展示节点效果预览，也不提供其他独立调整入口。

#### 12.5.1 创建调整建议

调整建议至少包含：

- 用户可读的调整说明
- 建议改变的策略字段
- 调整理由及其对话证据
- 当前正式 `strategy_version`
- 发起建议的问 AI session id
- 发起建议时用户所在的 `section_id + segment`
- proposal 版本和状态

对话界面必须明确说明哪些行为会改变，例如导读更关注概念关系、减少常识性注释或在节后
增加复述检查。不能只显示“优化处理方式”之类的笼统描述。

工具调用成功只创建待确认建议，不改变当前正式策略。同一用户书籍同一时间只允许有一个活动
中的 proposal。

#### 12.5.2 确认或反馈

确认卡归属于发起它的问 AI 会话，并紧跟触发工具调用的 AI 回复展示，不能固定在对话顶部。
确认卡至少提供：

- “确认调整”
- “反馈”

用户点击“反馈”时不进入独立表单或预览页，而是在原问 AI 会话中继续输入。Agent 根据反馈
修订同一 proposal，并再次调用处理方式调整工具。工具成功后，在本次 AI 回复后展示更新后的
确认卡；旧候选草稿标记为 `superseded`，不能继续确认。

正式阅读阶段的处理方式反馈不使用初次试读流程的 `adjustment_count`，首发不限制反馈次数。
用户明确表示不需要调整时，取消当前 proposal，正式策略保持不变。

#### 12.5.3 确认生效

用户点击“确认调整”时，服务端必须原子校验：

- proposal 仍是当前活动建议
- 用户书籍仍处于 `active_reading`
- 候选 `strategy_draft_version` 仍是 proposal 的当前草稿

校验通过后：

1. 候选草稿提升为新的正式 `strategy_version`。
2. proposal 标记为 `adopted` 并关联新的正式策略版本。
3. 当前正在阅读的节点和其他已读节点继续使用原有策略版本和增强内容。
4. 后续请求未读节点时使用新的当前策略；已有旧策略结果仍按原策略保存，但阅读器只选择当前策略
   对应的未读节点内容。
5. 历史问答、旧策略和旧节点内容全部保留。
6. 界面显示非阻塞成功提示，例如“处理方式已更新，后续内容将按新方式生成”。

重复确认请求必须幂等。建议已经失效或确认事务失败时，当前正式策略和节点内容保持不变，并允许
用户继续在原问 AI 会话中讨论。首发不追踪和拦截所有同时运行的旧任务，也不执行跨对象发布门禁。

### 12.6 使用限制

首发不设置产品层面的提问次数、追问轮数或每日额度。实现仍需具备：

- 单次模型调用超时
- 失败重试上限
- 单会话上下文压缩或截断
- 同一用户异常并发保护

这些属于系统稳定性保护，不在界面展示为商业额度。

## 13. 最近删除与恢复

### 13.1 最近删除页面

展示 30 天内删除的用户书籍，包括：

- 封面、书名和作者
- 删除时间
- 自动清理日期或剩余天数
- 恢复
- 立即永久删除；首发可不提供，若不提供则仅等待自动清理

### 13.2 恢复

恢复时：

- 清除 `deleted_at` 和 `purge_after`
- 恢复删除前 `workflow_status`
- 回到普通书架
- 原状态依赖后台任务时重新入队
- 已失效或已取消任务不得直接当作成功结果复用

### 13.3 自动清理

30 天后清理该用户与本书相关的：

- 访谈记录
- 本书画像
- 简报和策略版本
- 试读结果
- 节点增强结果
- 阅读进度、划线和笔记
- 有效阅读 session 和按书阅读统计
- 问 AI 会话

共享规范化书籍包不随单个用户删除。

为了避免删除一本书后用户的累计阅读时长和连续阅读天数倒退，系统保留不再关联具体书籍的全局
每日有效阅读时长汇总。该汇总只保存用户、自然日和有效秒数，不保留书名、阅读位置、笔记、原文
或其他按书详情。

## 14. 状态模型

### 14.1 共享书籍状态

```text
uploaded
  -> fingerprinting
  -> queued
  -> normalizing
  -> validating
  -> indexing
  -> analyzing
  -> ready
```

`normalizing`、`validating`、`indexing` 和 `analyzing` 均可进入 `failed`。重试创建新 job，但
共享书籍记录仍关联同一 EPUB 哈希。

### 14.2 用户书籍状态

```text
on_shelf
  -> interviewing
  -> strategy_review
  -> trial_generating
  -> trial_review
  -> active_reading

trial_generating
  -> trial_generation_failed
  -> trial_generating
```

软删除不进入状态链，通过 `deleted_at` 覆盖任意状态。

服务端必须校验所有状态迁移。前端路由不能直接改变业务状态。

### 14.3 节点增强状态

```text
not_requested -> queued -> generating -> ready
                                  \-> failed -> retrying
```

生成失败不影响节点原文可读。

### 14.4 试读版本状态

建议最小状态：

```text
draft -> generating -> ready -> published -> adopted
                    \-> failed
任意未 adopted 状态 -> superseded
```

任何发布和采用操作必须同时校验 `trial_revision` 与 `strategy_draft_version`。

### 14.5 正式阅读策略调整状态

调整建议建议使用以下状态：

```text
proposed -> pending_confirmation -> adopted
pending_confirmation -> proposed
proposed | pending_confirmation -> cancelled
任意未 adopted 状态 -> superseded
```

proposal 状态不是用户书籍主工作流状态。用户书籍在整个过程中保持 `active_reading`，当前正式
策略和阅读内容持续可用。

## 15. 数据要求

### 15.1 事实来源

业务数据库是以下状态的唯一事实来源：

- 用户和登录状态
- 画像当前版本
- 用户书籍工作流状态
- 当前策略草稿和正式策略
- 试读 revision、反馈次数和确认状态
- 节点生成状态
- 阅读进度、已读节点、有效阅读 session 和每日/按书统计
- 问答会话及策略调整建议
- 正式阅读策略调整的 proposal、候选草稿、确认和反馈状态

Agent session 只保存对话和工具轨迹，不能代替业务数据。

### 15.2 结果归属

用户可见的 AI 结果必须保存能够确定其归属和用途的直接业务标识，例如 user book、trial revision、
strategy 或问 AI session，以及 `generation_scope`。模型和 prompt 使用普通配置标识记录，便于定位问题。

首发不保存所有输入对象的完整版本快照，也不在异步结果写入前执行跨画像、策略、prompt 和模型配置
的通用版本比较。读取试读和正式阅读内容时，根据当前 trial、当前策略和 `generation_scope` 选择结果。

### 15.3 幂等性

以下操作必须幂等：

- 新用户添加预置书籍
- 相同哈希 EPUB 创建或复用共享任务
- 提交同一访谈答案
- 保存同一策略草稿版本
- 创建、发布和采用同一试读 revision
- 创建同一节点生成任务
- 最终采用处理方式
- 创建、修订和确认同一正式阅读策略调整建议
- 恢复同一本已删除书籍
- 提交同一个阅读时间区间或 heartbeat

## 16. 基础日志与产品事件

首发使用结构化应用日志和少量产品事件，不建设独立日志查看后台、完整审计仓库或通用 Agent
工具轨迹查询系统。

### 16.1 通用 AI 调用日志

每次 Agent run 或普通模型调用记录：

- 时间和请求 id
- 适用的用户书籍、共享书籍、节点或任务 id
- AI 功能类型
- provider、模型和 prompt 配置标识；不记录 API key
- 开始、结束、耗时和最终状态
- token 用量；provider 能够返回时记录
- 错误类型和经过脱敏的错误摘要
- 产物 id 或 artifact reference

### 16.2 Agent 工具日志

Agent 工具调用写入普通结构化日志，记录：

- Agent 类型和 session id
- 工具名
- 调用结果、耗时和错误摘要

不得记录 API key、OAuth token、对象存储密钥、完整宿主机路径或不必要的整本原文。

只有长期画像更新、处理方式调整等确实改变业务数据的工具，才由对应业务表保存变更结果和来源；
不为所有工具调用建立独立审计记录。

### 16.3 产品事件

为验证 MVP 核心假设，记录：

- 登录成功
- 打开书架中的书
- 开始和完成本书访谈
- 确认处理方式并生成试读
- 查看完三个试读片段并最终采用
- 进入正式阅读
- 正式阅读期间的有效阅读时间和问 AI 行为

这些事件可以写入简单产品事件表或现有业务记录；首发不要求统一事件平台、严格投递协议或完整操作
轨迹。

## 17. 错误与恢复原则

### 17.1 用户输入错误

表单校验应在字段附近显示中文错误，并保留用户已输入内容。重复提交期间禁用主按钮。

### 17.2 网络中断

- 用户答案、反馈、笔记和问题提交成功后由服务端返回稳定 id。
- 请求结果不确定时，客户端使用幂等 key 重试，不能创建重复版本。
- 阅读进度可以在网络恢复后补传，以服务端时间和客户端位置事件综合处理冲突。

### 17.3 AI 失败

- 访谈提问失败：停留当前问题并允许重试。
- 策略生成失败：保留访谈结果并允许重试。
- 试读失败：整轮失败，不展示部分结果。
- 正式节点增强失败：显示纯原文，后台重试。
- 问 AI 失败：保留用户问题，允许重试同一回答，不创建新会话。
- 处理方式调整工具失败：不展示可确认卡，保留当前正式策略和节点内容，允许 Agent 重试或继续回答。
- 策略调整确认失败：不部分生效，当前策略、已读节点和未读节点必须全部保持确认前状态。

### 17.4 多页面和多设备

首发不实现通用的多标签页并发协议。普通资料、笔记和进度采用最后一次成功写入的结果；访谈确认、
试读采用和策略调整确认等关键操作只在数据库事务中检查当前业务状态。操作因状态已经变化而失败时，
客户端重新读取当前状态并展示对应页面。

## 18. 安全与内容边界

- 所有用户资源按 user id 校验归属。
- Agent 不接触数据库连接和基础设施密钥。
- 规范化脚本在无网络沙箱内运行。
- EPUB 中的脚本、样式、iframe、object 等危险内容按 `nb-1.0` 清理。
- Markdown 形式的 AI 内容必须经过安全渲染，禁止任意 HTML 和脚本执行。
- 媒体资源通过受控路由或临时 URL 加载，HTML 中不得固化过期签名 URL。
- 用户上传的文件名、书名和作者字段在展示时必须转义。
- 日志中的用户文本和原文片段应遵循最小必要原则，不能为了调试无界复制整本书。

## 19. 页面级验收标准

### 19.1 登录与画像

- [ ] 新用户只能通过 Google OAuth 登录。
- [ ] 新用户完成固定三题初始画像问卷后进入书架。
- [ ] 刷新页面不会丢失已提交答案。
- [ ] 预置书籍只加入一次。

### 19.2 书架与上传

- [ ] 书架状态与点击目标符合状态映射。
- [ ] 用户可以上传 EPUB 并看到可恢复的处理阶段。
- [ ] 相同 EPUB 哈希复用 `ready` 或进行中的共享任务。
- [ ] 共享书籍未 ready 时不能阅读或开始访谈。
- [ ] 失败后可以重试或上传其他版本。

### 19.3 访谈与策略

- [ ] 每次只展示一个问题。
- [ ] 每个问题有 2–5 个选项并允许文字补充。
- [ ] 总问题数不超过 7 个且不能跳过整个访谈。
- [ ] 完成后生成本书画像、简报和策略草稿。
- [ ] 未经第一次确认不能生成试读。
- [ ] 处理方式页面不展示复杂设置。

### 19.4 试读

- [ ] Agent 选择三个不重叠、可精确定位的连续片段。
- [ ] 三个片段全部来自具有裁读资格的正文节点。
- [ ] 试读和正式阅读共用生成与校验实现。
- [ ] 三类辅助内容按策略和实际价值选择性生成。
- [ ] 三类内容全部为空的片段不能发布。
- [ ] 三个片段逐个查看并显示章节路径。
- [ ] 三个片段全部打开后才允许最终采用。
- [ ] 任一片段永久失败时不展示部分结果。
- [ ] 新反馈立即使整个旧 revision 失效并返回策略确认。
- [ ] 两阶段反馈合计最多 5 次。
- [ ] 未经最终确认不能创建正式策略或进入阅读。

### 19.5 阅读器

- [ ] 原文在任何增强内容状态下都可阅读。
- [ ] 导读和节后助读仅在有内容时显示，默认展开且可收起。
- [ ] 原书注和裁读注在原文位置触发并清晰区分。
- [ ] 阅读器提供字号、行距、内容宽度和主题设置。
- [ ] 进度只计算原文。
- [ ] 保存并跨设备恢复到 block + offset。
- [ ] 有效停留后节点永久标记为已读。
- [ ] 当前及后续三个节点持续预生成。
- [ ] 非正文或被排除的节点不创建裁读生成任务，但原文、划线、笔记和问 AI 正常可用。
- [ ] 产品不提供书签和独立笔记；笔记只能在原文划线后创建。
- [ ] 一个划线可以单独保存，也可以关联一条可编辑、可删除的笔记。
- [ ] 页面后台或持续空闲期间不累计有效阅读时间。
- [ ] 阅读统计展示今日、本周、累计和连续阅读天数。
- [ ] 按书展示累计阅读时长、进度、最近阅读时间和预计剩余阅读时间。
- [ ] 阅读数据不足时使用默认速度显示近似剩余时间，积累足够数据后改用该书个人速度。
- [ ] 回读和目录跳转不会被重复计算为向前阅读的原文字符量。
- [ ] 只有正常向前阅读原文的时间和字符进入该书个人阅读速度样本。

### 19.6 问 AI

- [ ] 划线和当前屏幕两种入口都可用。
- [ ] 每个新问题创建独立会话，追问复用该会话。
- [ ] 可以按需搜索全书和未读内容。
- [ ] 失败重试不会重复创建问题会话。
- [ ] 可以返回发起问题时的原文位置。
- [ ] 自动更新长期画像后显示非阻塞提示并记录证据。
- [ ] 本书处理方式只能通过问 AI 中的待确认建议修改。
- [ ] Agent 判断需要调整处理方式时调用工具创建待确认建议，不能直接修改正式策略。
- [ ] 确认卡紧跟触发工具调用的 AI 回复展示，并归属于该问 AI 会话。
- [ ] 用户点击“反馈”后在原会话继续输入；Agent 修订同一 proposal 后重新提交确认卡。
- [ ] 未经用户明确确认，工具调用和候选草稿都不能修改正式策略。
- [ ] 用户确认后，当前节点和已读节点保持旧版本，不被后台替换。
- [ ] 未读节点只展示当前策略对应的结果；旧策略任务即使完成也不会被读取为当前内容。
- [ ] 工具失败、取消建议或确认失败不影响当前正式阅读。

### 19.7 删除恢复

- [ ] 删除不改变原 `workflow_status`。
- [ ] 删除后从书架消失并出现在最近删除。
- [ ] 30 天内恢复到删除前状态。
- [ ] 单个用户删除不影响共享书籍包和其他用户。
- [ ] 永久删除一本书后，全局累计阅读时长和连续阅读天数不倒退，且保留汇总不再关联该书。

## 20. 端到端完成标准

MVP 可以被认定为完成，必须至少通过以下真实流程：

1. 新用户完成 Google 登录和初始画像，在书架看到预置书籍。
2. 用户打开预置书籍，完成不超过 7 问的访谈。
3. 用户查看简报，对处理方式提交一次反馈，再确认生成试读。
4. 系统选择并生成三个试读片段，三种辅助内容按策略选择性出现。
5. 用户逐个打开三个片段并最终采用处理方式。
6. 用户进入阅读器；即使首个节点增强仍在生成，也能立即阅读原文。
7. 用户修改阅读设置、滚动阅读、离开并在另一设备恢复到原位置。
8. 用户打开原书注和裁读注，并能明确区分来源。
9. 用户划线并添加笔记，离开后重新打开仍能回到对应原文位置。
10. 用户阅读一段时间后，在统计视图看到今日、本周、累计、连续天数和该书预计剩余时间。
11. 用户划线提问、继续追问并返回原始划线位置。
12. 问 AI 产生长期画像更新时，用户看到提示，画像记录保存本次变更来源。
13. Agent 在问 AI 中判断需要调整处理方式，调用工具后在对应回复下展示确认卡。用户先反馈
    一次，Agent 修订同一建议并重新提交；用户确认后，当前和已读节点保持不变，未读节点按新
    策略重新生成。
14. 用户删除书籍并从最近删除恢复，回到删除前流程状态。

同时必须通过以下技术验收：

- 规范化书籍符合 `nb-1.0` 且带源 EPUB 通过 `nb_check`。
- manifest 可以确定性重建，阅读节点和 block 位置稳定。
- 所有 AI 注释锚点可以精确解析。
- 用户可见 AI 产物能够关联到所属用户书籍、试读或正式策略，并记录模型和 prompt 配置标识。
- 读取试读和正式阅读内容时只选择当前 trial 或当前策略对应的结果。

## 21. 允许实现方自行决定的事项

只要不改变本文产品行为，以下事项可由实现方根据现有代码和部署条件决定：

- Web 框架、组件库和路由组织
- PostgreSQL 表名和字段名
- 对象存储供应商和本地存储实现
- 任务队列及 Worker 部署方式
- 进度轮询、SSE 或 WebSocket
- 全文索引、向量检索或混合检索
- 各 Agent 的最大 turn、超时和具体重试次数
- 试读片段建议长度
- 阅读进度保存的节流间隔
- 阅读 heartbeat 周期和空闲阈值
- 默认阅读速度、个人速度启用阈值和异常速度过滤参数
- 页面具体视觉风格和动画
- 默认模型和不同任务的模型覆盖配置

这些参数必须配置化，且不得绕过状态、版本、权限、原文不可变、初始流程的两次确认，以及
正式阅读策略调整必须由 Agent 工具创建待确认建议并经用户明确确认等核心规则。
