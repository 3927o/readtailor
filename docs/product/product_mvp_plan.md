# 裁读 ReadTailor 产品化梳理与 MVP 方案

**版本**：v0.2
**日期**：2026-07-13
**状态**：产品方向参考；用户可见行为以 `product_prd.md` 为准

Agent 的职责、工具、权限和完成条件详见 [`agent_design.md`](../architecture/agent_design.md)。
阅读节点、精确锚点、节点内容生成和阅读器数据契约详见

本文档整理裁读从实验原型走向网页产品时的产品流程、Agent 分工、核心状态和
MVP 边界。它不复用 `src/*.rs` 的旧实现，也不以旧 CLI 流程作为技术约束。

书籍规范化产物仍以 [`normalized_book_spec.md`](../contracts/normalized_book_spec.md) 的
`nb-1.0` 契约为准，最终验收使用 `tools/nb_check.py` 和 `tools/nb_linter.py`。

## 1. 产品目标

裁读不是摘要工具，也不替用户阅读。它通过了解用户的背景、目的和阅读阻碍，
为一本具体的书生成个性化陪读方式，降低阅读过程中的理解阻力，帮助用户真正读完。

首发产品是一个支持登录、书架、EPUB 上传、个性化访谈、试读确认和在线阅读的网页应用。

首发支持的裁读内容包括：

- 个性化读前简报
- 阅读节点开始前的导读
- 锚定到原文位置的 AI 注释
- 阅读节点结束后的助读
- 划线问 AI
- 基于当前屏幕的直接问 AI

其中“章节导读”和“节后助读”的展示形态可以相同，内容由本书处理方式决定，
不预先限制为摘要、问题或预告等固定类型。

**原文不可变原则**：所有裁读内容都是附加层。系统不得因为生成导读、注释或助读而
改写、删减或替换原书正文。

## 2. 核心对象

### 2.1 规范化书籍包

由某一个确定版本的 EPUB 生成，可在不同用户之间复用，不包含任何用户数据。

最小产物：

- 源 EPUB 的文件指纹
- `normalize.py`
- `book.normalized.html`
- `reading_manifest.json`
- 规范化报告
- `nb-1.0` 校验报告
- 图片及其他必要资源

多媒体资源保存在书籍包的 `assets/` 目录，规范化 HTML 只使用 `assets/...` 相对路径。
发布时整个书籍包按 EPUB 哈希写入统一存储接口；生产环境可使用任意 S3 兼容对象存储，
小型服务器和本地开发可使用本地文件系统。HTML 中不得固化会过期的签名 URL。

首发按 EPUB 文件 SHA-256 精确复用。不同文件即使内容相近，也先视为不同书籍版本。

书名、作者数组、语言、封面路径、标识符、出版社、出版日期、源文件名和 EPUB SHA-256 等
结构化元数据保存在共享书籍数据库记录中，供书架和业务查询使用。首发不额外创建
`book_metadata.json`；规范化 HTML 的 head 和 reading manifest 的 document 字段只保存由同一
次提取结果生成的必要快照，不能成为独立可编辑来源。

### 2.2 长期用户画像

描述相对稳定、可以跨书复用的阅读特征，例如：

- 大致知识背景
- 偏好的解释方式

最小结构只包含 `summary`、自由文字 `knowledge` 条目和自由文字
`explanation_preferences` 条目。不保存兴趣、阅读习惯或普遍阻碍字段。

注册时通过固定三题问卷生成简版画像，分别询问知识背景、解释方式和默认背景补充深度；固定
选项由程序映射，自由文字只做可失败降级的受限提取。之后每本书的访谈和问 AI 都可以在后台
更新长期画像，不需要用户逐次确认。

### 2.3 本书用户画像

描述用户为什么读这本书、对它已经知道什么、希望读到什么程度，以及最可能在哪里放弃。
每本书都必须单独访谈。

### 2.4 本书处理方式

由本书用户画像、长期画像和书籍结构共同生成，是后续内容生产的唯一依据。

它既要有用户可读的自然语言说明，也要有程序可消费的结构化版本。用户不能直接编辑
复杂设置，只能在处理方式确认页或试读页提交自由文字反馈让 AI 调整，最终明确点击
“采用这个处理方式并开始阅读”。

### 2.5 阅读节点

由程序根据规范化 section 树和正文顺序确定，用于模型调用、懒加载、进度、目录定位和
问 AI 上下文。共享的 `reading_manifest.json` 保存节点顺序和位置索引，不复制原文。

manifest 同时按版本化确定性规则记录节点是否具有裁读资格。首发只有 `bodymatter` 中由
`chapter`、`section` 或 `subsection` 所有、且包含可处理文本的节点具有资格；前言、导论、
附录、后记等非正文结构节点不生成导读、裁读注或节后助读，但仍可阅读、划线和问 AI。

首发不再按节点设置不同处理强度。导读、裁读注和节后助读是全书统一支持的三类内容；
某项对当前节点没有价值时，内容生成模型可以返回空。

## 3. 完整用户流程

### 3.1 注册与书架

1. 用户使用 Google OAuth 注册或登录。
2. 完成固定三题的简版长期画像问卷，问题覆盖知识背景、偏好的解释方式和默认背景补充深度；
   固定选项由程序映射为画像，自由文字只做可失败降级的受限提取。
3. 系统把预置书籍直接加入新用户书架。
4. 用户也可以上传自己的 EPUB。

没有公共书库、发现页或书籍市场。预置书目先通过管理脚本维护。

用户在访谈、处理方式确认、试读生成或试读查看期间返回书架或稍后重新打开书籍时，恢复离开
前的当前阶段和版本，不自动重置，也不提供“重新开始整本书配置”按钮。最终采用试读对应的
处理方式前，用户不能进入完整阅读器或绕过流程阅读整本纯原文。

### 3.2 EPUB 上传与规范化

1. 服务计算 EPUB 文件哈希。
2. 若已存在 ready 的规范化书籍包，直接复用其 HTML、assets、reading manifest 和 book profile，
   不重复运行后续规范化与 indexing 步骤。
3. 若未命中，创建规范化任务并执行以下步骤。
4. Coding Agent 阅读 EPUB 解包内容、`nb-1.0` 规范和校验结果。
5. Agent 为这一本书编写并迭代 `normalize.py`。
6. 脚本输出结构化 HTML 后，使用源 EPUB 作为 baseline 运行双层校验。
7. 校验通过后发布规范化核心产物；失败则保留报告并进入失败状态。
8. 程序从规范化 HTML 确定性生成并验证 `reading_manifest.json`。
9. 包级校验确认所有 `assets/...` 资源存在、路径安全且与 HTML 引用一致。
10. 书籍分析 Agent 基于只读规范化内容和 reading manifest 生成紧凑的共享
    `book_profile.json`，只包含全书级分析和一个小型试读候选池，不生成覆盖全部节点的摘要。
11. 上述步骤全部通过后书籍包进入 ready，才可用于访谈、试读选择和节点内容生成。

规范化是异步任务。网页展示站内进度，首发不发送邮件通知。

无法解包、无法规范化、加密或 DRM 等情况不做单独产品分支，统一进入规范化失败状态。
用户看到“该版本暂时无法处理”，可以重新上传其他版本或手动重试；首发不提供人工处理入口。

### 3.3 每本书的访谈

1. 用户打开一本尚未配置的书。
2. Agent 根据书籍信息、长期画像和已有回答逐题提问。
3. 问题以选项为主，同时允许自由文字补充。
4. 总问题数不超过 7 个；信息足够时允许提前结束。
5. 访谈产出本书用户画像，并可以顺带更新长期画像。

### 3.4 读前简报与处理方式确认

Agent 认为访谈信息足够后，同时生成个性化读前简报、用户可读的文字处理方式和对应的
结构化策略草稿。读前简报应帮助用户理解：

- 这本书整体在做什么
- 全书大致如何展开
- 用户在阅读前需要具备或临时补齐什么
- 对该用户而言，哪些部分最值得关注
- 哪些困难是正常的，应当如何进入

页面先展示读前简报和文字处理方式，不生成试读片段。用户可以提交自由文字反馈，要求
重新生成文字处理方式和结构化策略草稿；也可以明确点击“处理方式没问题，生成试读”。
未经用户确认的策略草稿不能用于试读。

进入正式阅读后，用户仍可以从目录或书籍菜单重新打开读前简报和当前处理方式。

### 3.5 试读片段与最终确认

1. 用户确认文字处理方式后，Agent 从全书选择三个具有代表性的连续原文片段，而不是机械
   选择前三节。每个片段必须位于一个具有裁读资格的阅读节点内，并以
   `section_id + segment + block range` 精确定位，不能跨阅读节点。
2. 三个片段原则上覆盖进入门槛、典型内容和高难度内容。
3. 系统按已确认用于试读的策略草稿生成三个试读结果。试读片段必须调用与正式阅读节点完全
   相同的内容生成器、固定生成脚本、输出 schema、锚点校验和失败重试流程；不得另做一套
   简化、美化或演示专用逻辑。业务差异只允许是输入原文范围和策略版本门禁。
4. 策略确认后页面立即显示三个固定槽位；Agent 选出的原文逐段出现，revision 创建后每段按
   pending/generating/ready/failed 独立更新，ready 段立即原地加入辅助内容。
5. 用户逐个查看三个片段；只有当前 revision 已发布且 3/3 ready，才显示或启用最终采用操作。
6. 用户认为试读有问题时提交自由文字反馈，旧试读在新策略 operation 成功前保持有效；成功后才
   原子 supersede 并返回处理方式确认页，失败时恢复旧试读和反馈。
7. Agent 根据试读反馈重新生成文字处理方式和结构化策略草稿。
8. 用户必须重新确认处理方式，系统才重新生成三个片段。
9. 用户明确点击“采用这个处理方式并开始阅读”后，策略才正式生效。
10. 三个片段任务按正式内容生成规则分别重试。任一任务耗尽重试仍失败时，整轮进入
    `trial_generation_failed`，禁止反馈和最终确认，但保留三个原文和已完成段。用户技术重试时精确
    复制失败 revision 的三个切片创建新 revision，不重新调用选段 Agent，也不增加反馈次数。

处理方式阶段反馈和试读阶段反馈合计最多 5 次，初始版本不计数。每次反馈都生成新的策略
草稿版本。达到上限后不再接受新反馈，但仍允许用户确认最后的策略草稿、生成最后一轮
试读，并最终采用或退出本书流程。达到上限后，处理方式页和试读页必须隐藏反馈输入框、
反馈按钮和其他重新调整入口；页面主体只保留当前阶段的确认操作。

新的处理方式或试读反馈只有在 operation 最终事务成功后才切换 current pointer；此前旧草稿或
`trial_revision` 保持权威。界面使用精确版本缓存，迟到的 GET/SSE 只能更新所属版本，不能覆盖当前
pointer。发布试读和最终采用时必须同时校验 `trial_revision` 与 `strategy_draft_version` 都是当前
版本，避免用户确认到已经过期的处理方式。

处理方式页面只展示自然语言说明，不做复杂设置面板。试读页逐个展示三个片段及最终采用入口。

### 3.6 正式阅读与提前生成

用户最终采用试读对应的处理方式后，不等待整本书生成完毕：

1. 第一个正式阅读节点的原文可加载后即可进入阅读器，不等待裁读内容完成。
2. 后台始终保证当前节点及后续三个阅读节点处于已生成或生成中状态。
3. 用户跳转到其他节点时，提高目标节点的处理优先级。
4. 某个节点生成失败时，用户仍然可以立即阅读纯原文，后台继续重试。
5. 原书脚注、尾注必须完整保留，并与 AI 生成的“裁读注”清晰区分。

阅读进度、已读节点和问 AI 记录登录后跨设备同步。

正式阅读器采用连续滚动。用户可以通过目录任意跳转到全书任何位置，包括尚未读到的后文，
不设置顺序解锁；目标节点裁读内容未完成时先显示纯原文。

阅读器呈现规则：

- 章节导读和节后助读分别作为原文前后的独立内容块，默认展开，允许用户收起。
- 原书注和裁读注都通过正文内标记触发，点击后在对应原文位置展开，并明确标注来源。
- 全书进度按原书正文位置计算，不把导读、注释和助读计入进度。
- 界面同时显示全书百分比和当前目录节点。

### 3.7 问 AI

支持两种入口：

- 划线提问：初始上下文是划线内容、所在段落和当前阅读节点。
- 直接提问：没有划线时，初始上下文改为当前屏幕内容和当前阅读节点。

两种方式都允许按需检索全书，包括尚未读到的后续内容，不做默认防剧透限制。

首发问 AI 不强制展示可点击的原文引用。

每一个新问题创建一个独立会话。用户可以围绕该问题继续追问，但不会形成一本书共用的
无限长聊天。问答产生的信息可以自动更新长期画像；若要修改本书处理方式，Agent 必须调用工具
创建待确认建议。确认卡展示在触发工具调用的 AI 回复之后，用户可以直接确认，也可以在原会话中
反馈并让 Agent 修订同一建议。用户明确确认后，新的策略版本才正式生效。

采用调整后，当前节点和其他已读节点保留旧版本，尚未阅读的节点使用新策略重新生成。历史问答、
旧策略和旧节点内容均保留，避免阅读记录被静默改写。

不提供独立的“调整本书处理方式”按钮。处理方式调整统一从问 AI 对话中发起。

### 3.8 删除与恢复

软删除不是阅读流程的下一个业务状态，而是覆盖在任意用户书籍状态上的独立标记。用户书籍
记录保留原来的 `workflow_status`，另外保存 `deleted_at` 和 `purge_after`。

用户可以在访谈、处理方式确认、试读生成、试读确认或正式阅读期间从书架删除一本书：

- 书籍进入“最近删除”
- 暂停或取消该用户尚未完成的访谈、试读和章节生成任务；共享书籍规范化任务不受影响
- 30 天内可以恢复
- 恢复时清除删除标记，回到删除前的 `workflow_status`；原状态需要后台任务时重新入队
- 30 天后清理该用户与本书相关的访谈、处理方式、进度和问答记录
- 共享规范化书籍包不随单个用户删除

## 4. Agent 与模型分工

并非所有 AI 步骤都需要 Agent。首发应把需要探索、使用工具和反复修正的任务交给
Agent，把稳定、可批量执行的内容生成交给普通模型调用。

### 4.1 规范化 Coding Agent

使用 Pi Agent SDK。它是整个流程里唯一需要编写和执行代码的 Agent。

职责：

- 阅读 EPUB 文件结构和少量代表性内容
- 阅读 `nb-1.0` 规范
- 编写单书 `normalize.py`
- 运行脚本
- 运行 `nb_check.py --baseline <source.epub>`
- 根据结构和保真错误继续修改脚本
- 校验通过后提交规范化书籍包

模型配置必须支持任意 OpenAI 兼容接口，通过环境变量传入，默认可使用 DeepSeek。

Agent 不应获得无限制宿主机 shell。建议只提供列文件、读文件、搜索、写规范化脚本、
运行规范化脚本、运行校验和完成任务等有限工具。模型生成的脚本在 E2B 沙箱内运行。

### 4.2 访谈与处理方式 Agent

这是面向用户的连续 Agent，但不执行代码。

职责：

- 根据书籍和已有画像动态提问
- 判断是否已有足够信息
- 生成本书用户画像
- 更新长期画像
- 生成自然语言和结构化的本书处理方式
- 选择三个试读片段及其精确 block range
- 根据处理方式页或试读页的自由文字反馈修订处理方式

### 4.3 章节内容生成器

使用普通 OpenAI 兼容模型调用，默认 DeepSeek，不使用 Coding Agent。

试读片段和正式阅读节点共用这一套生成器。两者唯一的业务差异是输入原文范围和策略版本：
试读处理 Agent 选中的三个节点内连续 block range 并使用已批准的策略草稿，正式阅读处理完整
节点并使用最终生效策略。`generation_scope` 只用于范围、策略版本门禁和缓存隔离，不得触发
不同的 prompt、内容标准或输出结构。

输入至少包括：

- 当前生成范围的结构化 HTML；试读为节点内连续 block range，正式阅读为完整节点
- 原书相关注释
- 必要的上级标题，以及程序按固定上限截取的相邻可裁读节点原文
- 书籍概要
- 长期画像
- 本书用户画像
- 试读时使用用户已批准用于试读的 `strategy_draft_version`
- 正式阅读时使用最终生效的 `strategy_version`

输出是结构化裁读内容，不返回重写后的整章 HTML。程序按 `reading_manifest.json` 提供
block 文本并验证精确词句锚点。首发不使用视觉模型；图片只向模型提供原始 alt 和图注。

### 4.4 问 AI Agent

使用只读检索工具，按问题创建独立会话。

可用能力：

- 读取当前屏幕和当前节点
- 获取相邻节点
- 搜索全书
- 读取指定节点
- 读取原书注释
- 读取长期画像和本书处理方式

它不能直接修改本书处理方式。需要调整时，必须通过工具创建待确认的变更建议；用户可以直接
确认，或在原问 AI 会话中反馈并让 Agent 修订。只有用户明确确认后才创建正式策略版本，当前
版本不要求额外生成节点效果预览。

## 5. 关键结构化产物

首发至少需要以下稳定产物，不能只保存 Markdown 对话：

### 5.1 `reader_profile.json`

只保存用户可查看和逐条纠正的长期知识背景、解释偏好，以及由这些条目生成的简短总结。
不保存兴趣、阅读习惯或普遍阻碍字段。更新时间和版本由数据库记录处理。

### 5.2 `book_reader_profile.json`

保存用户对当前书的目的、基础、阻碍、投入预期和其他访谈结论。
更新时间和版本由数据库记录处理，不写进画像 JSON 本身。

### 5.3 `book_profile.json`

共享的紧凑书籍分析结果，包括全书概要、结构特点、核心问题、主要主题、阅读门槛、阅读
建议，以及约 9–15 个可供后续个性化选择的试读候选位置。候选池只保存节点引用、候选特征
和简短选择理由，不保存覆盖全部 reading manifest 节点的摘要。

书籍分析 Agent 不负责裁读资格；资格由 reading manifest 生成程序按版本化规则确定。首发
不要求完整的节点级 AI 分析索引。未来如确有需要，可以增加独立、可懒生成和缓存的
`node_analysis`，但它不能成为共享书籍包进入 ready 的前置条件。

### 5.4 `strategy.json`

至少包含：

- 处理目标和表达原则
- 导读应解决的问题
- 注释重点和明确不需要注释的内容
- 节后助读的目标
- 当前策略版本

同一结构分别保存为 `strategy_draft_version` 和正式 `strategy_version`。前者在第一次确认后
只能用于试读，后者只在用户最终采用试读后创建并用于正式阅读。

正式阅读期间，用户可以从书籍菜单查看当前本书处理方式，但不能直接编辑。历史策略版本
在系统内部保留，不作为首发用户界面的一部分。

### 5.5 `trial.json`

记录三个试读片段的 `section_id + segment + block range`、选择理由、逐段生成状态、所用
`strategy_draft_version`、`trial_revision`、两阶段共用的
`adjustment_count`、当前反馈阶段、试读版本状态、是否已批准用于试读，以及是否已最终采用。
`trial_revision` 在创建三个生成任务之前分配；三个任务和结果都必须携带该 revision。收到成功提交的新
反馈时整个 revision 标记为 superseded，不区分任务是否已经完成。

### 5.6 节点增强结果

每个节点分别保存导读、AI 注释和节后助读。AI 注释必须包含稳定的原文锚点，不能依赖
模型生成的模糊文本匹配。AI 注释和用户划线共用 `section + segment + block range` 位置模型。

### 5.7 `reading_manifest.json`

由规范化 HTML 确定性生成的共享阅读索引。它记录节点顺序、section、segment、标题层级和
正文统计，并记录完整 section 导航树、裁读资格及资格规则版本；不复制原文，可使用该书籍包
固定的算法版本重新生成。详细契约见 [`reading_contract.md`](../contracts/reading_contract.md)。

## 6. 核心状态

### 6.1 规范化书籍状态

```text
uploaded -> fingerprinting -> queued -> normalizing -> validating -> indexing -> analyzing -> ready
```

`normalizing`、`validating`、`indexing` 或 `analyzing` 任一阶段均可进入 `failed`。

### 6.2 用户书籍状态

```text
on_shelf -> interviewing -> strategy_review -> trial_generating
trial_generating -> trial_review -> active_reading
trial_generating -> trial_generation_failed -> trial_generating（重试）
```

`strategy_review` 中的确认只允许生成试读片段，不代表策略正式生效。用户在 `trial_review`
明确采用试读对应的处理方式后，才能进入 `active_reading`。

`trial_generating` 成功后进入 `trial_review`；任一任务耗尽重试后进入
`trial_generation_failed`，重试时回到 `trial_generating`。软删除不属于这条状态链，使用
`deleted_at` 独立覆盖当前状态。

### 6.3 阅读节点状态

```text
not_requested -> queued -> generating -> ready
                                   \-> failed -> retrying
```

`failed` 不影响原文可读。

## 8. 质量与完成标准

### 8.1 规范化质量

- 结构符合 `nb-1.0`
- 必须以源 EPUB 运行保真检查
- 可见字符召回率达到规范阈值
- 图片引用守恒
- 原书注释引用和正文数量可对账
- 源目录存在时，产物 TOC 可对账
- 未通过校验的书籍包不能进入用户阅读流程

### 8.2 个性化质量

- 访谈不超过 7 个问题
- 三个试读片段具有代表性、来自具有裁读资格的节点且不是机械前三节
- 文字处理方式未经确认不能生成试读片段
- 试读反馈必须回到处理方式确认，再重新生成三个片段
- 两个阶段的反馈合计最多允许 5 次
- 达到 5 次后隐藏全部反馈相关 UI，只保留当前阶段的确认 UI；仍允许确认最后策略、生成
  最后一轮试读并最终采用或退出
- 任一试读任务永久失败时不能发布或采用，但必须保留原文和已完成段并允许按原切片重试整轮
- 新反馈成功前旧 `trial_revision` 保持有效；成功后旧结果不能继续作为当前版本展示或确认
- 试读未经最终确认不能用于正式阅读
- 修改处理方式必须保留版本，不能静默覆盖正在阅读的内容

### 8.3 阅读可靠性

- 裁读内容失败时原文始终可用
- 原书注与裁读注不能混淆
- 当前节点之后三个节点持续预生成
- 划线发起的问 AI 会话可以回到原始划线位置
- 进度和问答可以跨设备恢复
- 裁读内容不得改写、删减或替换原文

## 10. 相对原始清单新增的关键事项

原始梳理主要覆盖了 EPUB 结构化、用户访谈、处理方式生成和章节懒加载。产品化还必须显式
补上以下事项：

1. 规范化书籍包与用户个性化数据分层，并支持按 EPUB 哈希复用。
2. Agent 生成脚本必须经过“执行、校验、修复”的闭环，而不是一次生成即接受。
3. 用户画像分为长期画像和本书画像，并规定更新权限。
4. 本书处理方式必须同时有自然语言和结构化版本。
5. 阅读节点由程序从规范化结构确定；本书处理方式不再维护逐节点强度配置。
6. 试读片段由 AI 选取并由程序保存精确范围；处理方式和试读反馈合计最多 5 次，试读反馈
   必须返回处理方式确认。
7. 用户先确认文字处理方式才能生成试读，再最终确认试读才能开始正式阅读。
8. 章节增强失败时原文必须继续可读。
9. 原书注释与 AI 注释必须并存并区分来源。
10. 问 AI 分为划线和当前屏幕两种入口，每个问题独立成会话。
11. 问答可自动更新长期画像，但修改本书处理方式必须创建待确认建议并由用户明确确认。
12. 阅读进度、问答和书架状态需要跨设备同步。
13. 删除采用 30 天软删除，用户数据与共享书籍包生命周期分离。
14. 导读和助读默认展开；原书注与裁读注均在原文位置触发展开。
15. 进度只计算原文，并同时展示全书百分比和当前目录节点。
16. 正式阅读中只通过问 AI 提议调整本书处理方式，不设置独立调整入口。
17. 长期画像允许用户查看和纠正；本书处理方式允许回看但不直接编辑。
